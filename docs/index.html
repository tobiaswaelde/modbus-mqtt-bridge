Modbus MQTT Bridge

A small Rust service that bridges Modbus TCP and MQTT with clear topic contracts, strong typing, and production-friendly Docker support.

What this service does: Poll Modbus points, publish raw values to MQTT state topics, and accept write commands via matching /set topics.

Getting Started

  1. Clone the repository.
  2. Edit config/config.yml with your broker and device settings.
  3. Start an MQTT broker.
  4. Run cargo run -- --config config/config.yml.
  5. Publish test commands to /set topics.
cargo run -- --config config/config.yml

Concepts

The runtime has three core loops:

  • MQTT event loop: maintains broker connection and receives writes.
  • Per-source polling loops: read configured Modbus points.
  • Write handler: validates and applies incoming /set commands.

Each source is isolated in its own task so one slow device does not block all others.

Architecture

The bridge sits between Modbus devices and MQTT consumers. It polls device registers, publishes state updates, and listens for write requests on matching /set topics.

Practical Recipes

These patterns cover the most common integrations without changing the bridge itself.

Home Assistant

Use MQTT entities that point at the bridge's state topic and write to the matching /set topic for controls. 

mqtt:
  sensor:
    - name: "Boiler Temperature"
      state_topic: "modbus/example-device/telemetry/boiler_temp"
      unit_of_measurement: "°C"
      device_class: temperature
      state_class: measurement

  switch:
    - name: "Pump Enable"
      state_topic: "modbus/example-device/controls/pump_enable"
      command_topic: "modbus/example-device/controls/pump_enable/set"
      payload_on: "true"
      payload_off: "false"

Node-RED

Subscribe to the state topic, transform values in a function node, then publish a command on the paired /set topic when automation conditions are met. 

[MQTT in] -> [Function: derive alarm] -> [Switch] -> [MQTT out]
   topic: modbus/example-device/telemetry/pressure
                                 topic: modbus/example-device/controls/fan_speed/set

Mosquitto

For quick validation, use Mosquitto clients to watch state changes and send test writes. This is the fastest way to confirm topic names and payload formats during setup. 

mosquitto_sub -h localhost -t 'modbus/+/+'
mosquitto_pub -h localhost -t 'modbus/example-device/controls/pump_enable/set' -m true

Topic Contract

With base_topic=modbus, source_id=example-device, and topic=telemetry/example_float:

  • State topic: modbus/example-device/telemetry/example_float
  • Set topic: modbus/example-device/telemetry/example_float/set

State payload example:

21.5

Write payload examples:

true
{"value": 18.0}

Configuration

Config can be YAML or JSON. Default path is config/config.yml. If the file does not exist, the service writes a starter template automatically. 

mqtt:
  host: localhost
  port: 1883
  client_id: modbus-mqtt-bridge
  base_topic: modbus

sources:
  - id: example-device
    host: 127.0.0.1
    port: 502
    unit_id: 1
    poll_interval_ms: 2000
    request_timeout_ms: 3000
    points:
      - name: example_float
        topic: telemetry/example_float
        address: 0
        kind: holding
        data_type: f32
        access: read_write

Supported enums

  • kind: coil, discrete_input, holding, input
  • data_type: bool, u16, i16, u32, i32, f32, string, raw_u16
  • access: read_only, write_only, read_write
  • encoding.byte_order / word_order: big or little

Docker

Build and start with Compose:

docker compose up --build

The image contains a built-in healthcheck and is optimized for size.

Troubleshooting

Use this checklist when the bridge starts but data does not move the way you expect.

  • Confirm the MQTT broker is reachable and the client credentials are correct.
  • Verify the Modbus host, unit ID, and register addresses against the device manual.
  • Check that the topic path exactly matches the configured source and point names.
  • Make sure the target point is writable before sending anything to a /set topic.
  • Watch logs for timeout, decoding, or payload validation errors.
Fast isolation tip: publish a known-good value with mosquitto_pub first. If the write path works there but not from your automation, the issue is likely upstream.

Operations

  • Run healthcheck manually: modbus-mqtt-bridge --healthcheck --config config/config.yml
  • Enable metrics with metrics.enabled: true and scrape http://<host>:9464/.
  • Enable JSON logs with logging.json: true.
  • Override log level with RUST_LOG when needed.
